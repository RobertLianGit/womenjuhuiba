# 「开始聚会吧」网页版 — 产品设计规格书 v1.0

**整理日期：** 2026-04-15
**版本：** v1.0
**状态：** 待用户确认

---

## 一、产品定位

**一句话：** 解决「朋友聚会怎么组织、费用怎么算」的全程工具，纯网页版，无需下载注册。

从创建活动、意愿收集、投票、方案确认、分段报名、记账分摊，一站式完成。

### 核心价值观

- **不收集任何个人隐私信息**
- 用户无需注册账号，访问即生成匿名身份
- 活动通过邀请链接/邀请码分享，受邀才能查看

## 二、用户身份体系

### 身份类型

| 类型 | 生成方式 | 用途 |
|------|---------|------|
| 匿名访客 | 首次访问自动生成 UUID | 正常使用，无需任何操作 |
| 已绑定邮箱用户 | 绑定邮箱后 | 跨浏览器/设备恢复身份 |

### 登录流程

**场景 1：新用户首次访问**
```
用户打开网页
  → 前端检测 localStorage 无 UUID
  → 自动生成 UUID（如 `8f14e45f-ceea-46...`）
  → 提示用户设置显示名（如"小明"或"神秘人-A3K2"）
  → 写入 users 集合
  → 完成，无感登录
```

**场景 2：返回访问（同一浏览器）**
```
用户打开网页
  → 读取 localStorage 的 UUID
  → 验证 users 集合中该 UUID 存在
  → 自动登录，跳转首页
```

**场景 3：换浏览器 / 清除数据 / 想换设备**
```
用户打开网页
  → 无 UUID，提示"设置显示名"或"恢复身份"
  → 用户选择"恢复身份"，输入邮箱
  → 后端发送验证码到邮箱
  → 用户输入验证码
  → 验证通过，生成新 UUID 并关联到该邮箱
  → 恢复所有关联的活动
```

### users 数据结构

```json
{
  "_id": "8f14e45f-ceea-46a9-a4e2-xxxxxxxxxxxx",
  "display_name": "小明",
  "recovery_email": "user@example.com",
  "email_verified": true,
  "created_at": "2026-04-12T10:00:00Z"
}
```

## 三、邀请机制

### 链接结构

**邀请链接格式：**
```
https://yourdomain.com/join/{activity_id}/{token}
```

| 参数 | 说明 |
|------|------|
| `activity_id` | 活动 UUID |
| `token` | 邀请令牌，关联发起者/参与者身份 |

### 邀请码（备用）

- 6 位大写字母+数字（如 `KLD9X2`）
- 活动页面上显示，备用方案
- 适用场景：朋友在另一个设备上手动输入

### 鉴权规则

| 链接类型 | 身份 | 权限 |
|---------|------|------|
| 发起者邀请链接 | 发起者 | 创建活动、管理活动、查看所有数据 |
| 参与者邀请链接 | 参与者 | 加入活动、填写意愿/投票/报名、查看活动 |

### 邀请令牌数据结构

```json
{
  "_id": "auto",
  "activity_id": "活动UUID",
  "token": "KLD9X2",
  "created_by": "发起者UUID",
  "role": "creator" | "participant",
  "used_by": ["user_uuid_1"],
  "max_uses": null,
  "created_at": "2026-04-12T10:00:00Z"
}
```

## 四、活动状态与流程

### 状态流转

```
created（创建）→ intention（意愿收集中）→ voting（投票中）→ plan_confirmed（方案已确认）→ registering（报名中）→ ongoing（进行中）→ settled（已结算）
```

### 核心流程

```
发起者创建活动（生成邀请链接）
    ↓
分享邀请链接给朋友
    ↓
朋友打开链接（设置显示名）→ 加入活动成为参与者
    ↓
发起者发起意愿收集（状态: intention）
    ↓
意愿收集 → 意愿汇总（发起者查看）
    ↓
发起者确认 → 发起投票（状态: voting）
    ↓
投票 → 投票结果（继承意愿阶段想去哪的选项）
    ↓
发起者确认方案 + 生成 Prompt（状态: plan_confirmed）→ 开启报名（状态: registering）
    ↓
参与者报名（写入 participants）→ 看板实时可见
    ↓
活动进行中（状态: ongoing）→ 结算（状态: settled）
```

## 五、功能详细说明

### 5.1 创建活动

- **入口：** Tab「我发起的」页面右下角的「发起新活动」按钮
- **必填：** 活动名称 + 活动描述
- **可选：** 大致时间（如"周末"、"下周"），具体时间在方案确认后由发起者填写
- **分段：** 在方案确认阶段由发起者自行设置（如"午餐/下午茶/晚餐"）
- **规模限制：** 最多 50 人
- **创建后：** 生成发起者邀请链接，可一键复制分享

### 5.2 意愿收集

#### 参与者视角

| 字段 | 规则 |
|------|------|
| 是否参与 | **必填** |
| 名字 | 参与时 **必填** |
| 想去哪 | 选填 |
| 参与时间段 | 选填（若活动分段，可选择参与哪几段） |

> 注意：发起者本人也需要主动填写意愿，不能默认自动参与。

#### 发起者视角——意愿汇总

- 总参与人数（名字列表）
- 各人想去的地方（文本汇总）
- 各人希望参与的时间段分布
- 未表态人员列表

#### 意愿截止时间

- 默认截止时间 **3 天**，可手动调整
- 发起者可提前开始投票，或手动延期

### 5.3 投票去哪（可跳过）

#### 意愿收集阶段结束后的选择

意愿收集结束后，发起者看到汇总结果，有两个选择：

- **选项 A：跳过投票，直接进方案确认**
  - 发起者根据意愿汇总直接拍板方案

- **选项 B：发起投票**
  - 投票的备选项**来自意愿收集阶段的 wants 数据**
  - 由发起者从汇总中提炼/筛选出 1-2 个核心方案作为投票选项
  - 发起者也可以新增自定义方案作为投票选项

#### 发起投票

- **只有发起者**可以发起投票
- 发起投票的前提：至少有 **1-2 个备选项**
- 若备选不足，发起者看到提示"请先汇总足够方案再发起投票"
- 发起者点击"发起投票"后，状态切换为 `voting`，投票正式开始

#### 投票阶段

- 参与者可**多选**：对多个方案投票（写入 votes 集合）
- 投票结果供发起者参考，**最终方案由发起者拍板确认**
- 投票设置截止时间，默认 **3 天**，可手动调整；到期自动关闭，发起者可**提前结束投票或延期**

### 5.4 方案确认

#### AI 生成方案

沿用 v1.0 标准 Prompt 方案：组织者一键复制 Prompt，粘贴到豆包/Kimi 等 AI 工具生成方案后，再手动复制粘贴回方案页，支持直接在方案页编辑。

> **数据来源：** Prompt 中的方案汇总根据实际情况填充：
> - **跳过了投票：** 使用意愿收集阶段的 wants 汇总
> - **经过投票：** 使用意愿收集阶段的 wants 汇总 + 投票结束后的各方案得票情况

#### Prompt 模板

```
【角色】你是一位资深活动策划专家。

【背景】
我正在组织一场朋友聚会活动，需要你帮我生成一份详细的活动方案。

【基础信息】
- 活动名称：{活动名称}
- 活动描述：{活动描述}

【参与者意愿汇总】
- 总参与人数：约 {人数} 人
- 想去的地方/活动形式（来自意愿收集阶段）：
{意愿收集阶段 wants 汇总}

【投票结果】（若无投票，此项可跳过）
{投票结束后各方案得票情况}
- 希望参与的时间段：
{汇总时间段分布}

【你的任务】
请生成一份完整的活动方案，包含：
1. 分段建议（几段、每段名称和时间）
2. 时间安排（整体时间线）
3. 地点推荐（2-3 个，含地址、人数、人均消费）
4. 活动流程（详细时间轴）
5. 费用预算参考
6. 注意事项
7. 备选方案

请用清晰的结构输出，语言轻松活泼，适合朋友圈分享。
```

#### 方案与报名联动

- 方案**支持分段**
- AI 方案中包含分段建议，发起者确认方案后，可一键采纳 AI 建议的分段，也可手动输入/调整
- 分段信息写入 `activities.scenes`
- 报名环节**按段展示**，参与者可选择参与哪几段

### 5.5 活动报名

#### 参与者视角

| 字段 | 规则 |
|------|------|
| 姓名 | 必填 |
| 参与哪几段 | 必填（若活动分段） |
| 人数 | 必填 |
| 备注 | 选填（如过敏、忌口、特殊需求等） |

报名不设截止时间，可随时修改或取消（在活动开始前）。

#### 发起者视角——实时看板

| 功能 | 说明 |
|------|------|
| 实时查看 | 随时看到每段活动的报名参与人列表 |
| 支持手动编辑 | 手动添加/删除参与者 |
| 临时参与人 | 与正常参与人混排，用角标区分"临" |
| 名单导出 | 可复制名单，方便发起者自己管理 |

### 5.6 记账结算

#### 数据联动

- 每场参与人数**直接来自报名看板上发起者确认后的最终数据**，自动带入
- 支持手动编辑

#### 结算方式

**方式 A：AA 均摊**
```
该场人均 = 该场总额 ÷ 该场实际参与人数
某人应付 = Σ（各场人均 × 该人参与的场次）
```

**方式 B：谁欠谁最优路径**
- 计算最少转账次数的结算方案
- 列出清晰的转账指引（如：A 给 B ¥50，B 欠 C ¥30 → 合并成 A 给 C ¥50）

#### 展示维度

- 按人维度：每人应付多少
- 按场次维度：每场多少人、花了多少、人均多少
- 本产品**不涉及任何真实资金流转**

## 六、页面结构

### 路由设计

```
/                           首页（Tab 切换：我发起的 / 我参与的）
/login                      设置显示名 / 身份恢复
/create                     创建活动
/activity/:id               活动详情
/activity/:id/intention     意愿收集
/activity/:id/intention-summary  意愿汇总
/activity/:id/vote          投票
/activity/:id/plan          方案页
/activity/:id/register      报名
/activity/:id/board         看板
/activity/:id/settle         结算
/join/:activityId/:token     邀请加入（无需登录，自动加入）
```

### 首页结构

- 顶部 Tab：「我发起的」/ 「我参与的」
- 活动列表按**距活动开始时间倒序**排列
- 显示倒计时（距活动开始还有 X 天/X 小时）
- 列表项显示：活动名称、状态标签、活动时间
- 右下角浮动「发起新活动」按钮

### 邀请加入页 `/join/:activityId/:token`

- 验证 token 有效性
- 已登录：直接进入活动
- 未登录：先设置显示名 → 加入活动
- 已在 participants 中：提示"已是参与者"

## 七、技术选型

| 层 | 技术 | 说明 |
|----|------|------|
| 前端框架 | Taro 4 + React + TypeScript | 编译 target: h5 |
| 样式 | SCSS（移动端优先） | 暖色主题，圆角卡片 |
| 前端部署 | 腾讯云 COS | 使用备案域名，微信内直接访问 |
| 后端 API | 腾讯云 SCF（云函数） | 纯 HTTP 接口，配置 CORS，不依赖微信环境 |
| 数据库 | 腾讯云 MySQL（或 PostgreSQL） | 你账号下的数据库，独立于微信生态 |
| 用户身份 | 邮箱注册 + UUID 匿名登录 | 不依赖微信 |

### 部署架构

```
[微信内置浏览器 / 任意浏览器]
    ↓ HTTPS 请求
[腾讯云 SCF]（纯 HTTP 接口，配置 CORS）
    ↓
[腾讯云 MySQL]（你账号下的数据库）
    ↓
返回 JSON
    ↑
[前端：腾讯云 COS + 备案域名]
```

### 为什么换掉微信云函数

上个版本卡在微信云函数：微信云函数依赖微信小程序上下文（openid、session），从浏览器直接调用会失败。改用腾讯云 SCF 后：
- 纯 HTTP 接口，任意浏览器都能调
- 在腾讯云后台直接上传，无需微信开发者工具
- CORS 配置简单，标准 HTTP 调用
